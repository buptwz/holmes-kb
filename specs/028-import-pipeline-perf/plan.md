# Implementation Plan: Import Pipeline Performance Optimization

**Branch**: `028-import-pipeline-perf` | **Date**: 2026-06-22 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `specs/028-import-pipeline-perf/spec.md`

## Summary

Five coordinated optimizations to the Holmes import pipeline that reduce latency for both single-document and batch imports while preserving output quality:

1. **Phase 3 per-KP isolation** — replace the single mega-context loop with a per-draft LLM loop; each KP gets a fresh `messages=[]` and a scoped source slice.
2. **Extractor parallelism** — run all KP Extractor agents concurrently in a `ThreadPoolExecutor`; each worker gets its own `DocumentCursor` copy.
3. **Rate-limit retry + jitter** — `_call_with_retry` helper in `provider/base.py` wraps both `AnthropicProvider` and `OpenAIProvider` SDK calls; `AnthropicProvider` gains full error-type parity with `OpenAIProvider`.
4. **`list_entries` process cache** — module-level dict in `store.py` eliminates the O(n) disk scan on every call; invalidated on any KB write.
5. **`--dir` concurrent batch + single commit** — `ThreadPoolExecutor` in CLI `--dir` branch; `ThreePhaseImportPipeline` gains `skip_git_commit` flag; single final commit after all files complete.

## Technical Context

**Language/Version**: Python 3.11+

**Primary Dependencies**: anthropic SDK, openai SDK, click (CLI), python-frontmatter, pytest

**Storage**: Markdown files with YAML frontmatter (KB entries); JSON sidecar files (evidence); local git repository

**Testing**: pytest (existing suite in `kb/tests/`); new unit tests for retry logic, cache invalidation, and parallel extraction

**Target Platform**: Linux server (Ubuntu 22.04+); developer workstation

**Project Type**: CLI tool + Python library

**Performance Goals**:
- Single 5-KP import: ≥30% wall-clock reduction vs baseline
- 4-KP Extractor phase: ≤1.5× slowest single-KP time (vs 4× serial)
- `--dir` 5-file batch: ≤2× slowest single-file import (vs 5× serial)
- `list_entries` I/O calls: ≥80% reduction per import run

**Constraints**:
- Zero quality regressions (all existing tests must continue to pass)
- No new external dependencies (use stdlib `concurrent.futures`, `time`, `random`)
- `extractor_concurrency=1` must be functionally identical to current serial behavior

**Scale/Scope**: KB sizes up to ~500 entries; documents with up to 10 knowledge points; `--dir` batches up to 50 files

## Implementation Order

The five optimizations are ordered by dependency and risk:

1. **Config extension** (`config.py`) — no risk, pure additive; needed by all other changes
2. **list_entries cache** (`store.py`, `pending.py`) — self-contained, easiest to test in isolation
3. **Retry logic** (`provider/base.py`, `anthropic_provider.py`, `openai_provider.py`) — self-contained, easy to unit-test with mocks
4. **Phase 3 per-KP isolation** (`pipeline.py`) — changes _run_extraction_loop; must not break existing tests
5. **Extractor parallelism** (`pipeline.py`) — builds on Phase 3 isolation; adds ThreadPoolExecutor
6. **--dir concurrent batch** (`cli.py`, `pipeline.py`) — builds on skip_git_commit from step 4; uses ThreadPoolExecutor

Each step has its own tests that can be run before proceeding to the next.
