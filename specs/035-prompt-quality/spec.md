# Feature Specification: Import Pipeline Prompt Quality Optimization

**Feature Branch**: `035-prompt-quality`

**Created**: 2026-06-18

**Status**: Draft

**Input**: User description: "Prompt 质量优化：参考业界最佳实践（如 Claude Code 的 system prompt 写法），对 Holmes KB 导入管线中的所有 LLM prompt 进行系统性改写。目标是让模型对指令的遵从更稳定、减少幻觉（尤其是 section 混用和字段捏造），同时降低 token 消耗。涉及范围：ImportAgentRunner system prompt、ExtractorAgent prompt、ReaderAgent prompt、ContentVerifier prompt、DocumentClassifier prompt、SkillAdvisor/SkillExecutor prompt。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Section Structure Integrity (Priority: P1)

A developer imports a post-mortem document describing an architecture decision. They expect
the resulting pending entry to be classified as `decision` type and contain only the sections
appropriate for that type (`## Context`, `## Decision`, `## Rationale`). Currently, the import
pipeline sometimes bleeds in pitfall-specific sections (`## Symptoms`, `## Root Cause`) into
non-pitfall entries, producing pending entries that must be manually rejected or corrected.

After this improvement, every pending entry generated by the import pipeline has sections that
match its declared type — every time, not just most of the time.

**Why this priority**: Section structure is the most visible quality failure. An entry with
wrong sections is structurally invalid and provides no value to KB consumers or the MCP agent.
This is the highest-frequency rejection reason for pending entries.

**Independent Test**: Import five documents of different types (pitfall, decision, process,
model, guideline). Inspect each generated pending entry. All five must contain only sections
valid for their declared type. No pitfall sections appear in non-pitfall entries.

**Acceptance Scenarios**:

1. **Given** a source document describing a technology selection rationale, **When** the import
   pipeline runs, **Then** the pending entry has type `decision` and contains `## Context` and
   `## Decision` but does NOT contain `## Symptoms` or `## Root Cause`.

2. **Given** a source document describing an operational runbook, **When** the import pipeline
   runs, **Then** the pending entry has type `process` and contains `## Steps` but does NOT
   contain `## Root Cause` or `## Resolution`.

3. **Given** a source document with ambiguous structure, **When** the import pipeline runs,
   **Then** the pipeline selects one type and consistently applies only that type's sections
   rather than mixing sections from multiple types.

### User Story 2 - Field Fabrication Prevention (Priority: P2)

A knowledge engineer confirms a pending entry submitted by the import pipeline, only to
discover that the `## Resolution` section contains steps that were never mentioned in the
source document. The content verifier was supposed to catch this. After this improvement,
the verifier reliably flags and removes fields that lack direct source text support, so
confirmed entries contain only facts extractable from the original document.

**Why this priority**: Fabricated content in confirmed entries undermines the entire KB's
trustworthiness. Once incorrect information is promoted to verified status, it actively
misleads future troubleshooters.

**Independent Test**: Import a short, tightly-scoped incident report. After import, compare
every field in the pending entry against the source document. Any field value that cannot
be traced to a verbatim or paraphrased passage in the source document should not appear.

**Acceptance Scenarios**:

1. **Given** a source document that describes symptoms and a root cause but provides no
   resolution steps, **When** the pipeline imports it, **Then** the `## Resolution` section
   in the pending entry is empty or absent — not filled with plausible-but-invented steps.

2. **Given** a source document with a specific version number mentioned once, **When** the
   pipeline imports it, **Then** the `tags` or body fields contain only that version number,
   not related version numbers the model might infer.

3. **Given** a malformed or very short source document, **When** the pipeline imports it,
   **Then** the verifier produces a warning listing every field it cleared for lacking source
   support, rather than silently accepting the draft.

### User Story 3 - Consistent Document Classification (Priority: P3)

A team uses Holmes to import a batch of 20 historical incident reports. They expect each
document to be correctly classified as `pitfall` type. After this improvement, the classifier
reliably identifies KB-worthy content and assigns the right type without requiring manual
`--type` overrides for straightforward documents.

**Why this priority**: Mis-classification forces users to rerun imports with `--type` flags
or manually reject entries. Reliable classification reduces operational friction for batch imports.

**Independent Test**: Run `holmes import --dir ./incidents/` on a directory of 10 known-format
incident reports. Inspect all generated pending entries. At least 9 of 10 must be correctly
typed without any `--type` override.

**Acceptance Scenarios**:

1. **Given** a batch of incident reports with `## Symptoms` / `## Root Cause` / `## Resolution`
   sections, **When** imported without `--type`, **Then** all entries are classified as `pitfall`.

2. **Given** a meeting notes document with no actionable troubleshooting content, **When**
   imported, **Then** the classifier marks it as non-KB-worthy and the pipeline skips it with
   a clear explanation.

3. **Given** a document that contains both architecture rationale and operational steps,
   **When** imported, **Then** the classifier picks the dominant type and creates one entry
   for each distinct knowledge point rather than producing a mixed-type entry.

---

### User Story 4 - Lower Token Cost Per Import (Priority: P4)

An operator running `holmes import --dir ./large-collection/` on a directory of 50 documents
notices that today's run completes in noticeably fewer API token units than a previous run
on equivalent documents. The quality of generated entries is unchanged or better.

**Why this priority**: Token reduction directly lowers operating cost and reduces latency for
large batch imports. This is a secondary concern behind accuracy, but becomes important at scale.

**Independent Test**: Import the same 5 reference documents before and after the change.
Compare total token usage reported by the provider. Token count must decrease without an
increase in the entry rejection rate.

**Acceptance Scenarios**:

1. **Given** the same source document imported twice (before and after the change), **When**
   comparing token usage, **Then** the after-change import uses fewer input tokens while
   producing an equivalent or better pending entry.

2. **Given** a verbose system prompt replaced by a structured, concise prompt, **When** the
   import pipeline runs, **Then** the shorter prompt does not increase the rate of structurally
   invalid entries.

---

### Edge Cases

- What happens when the source document is in Chinese and the prompt is in English? The
  rewritten prompts must still instruct the model to write KB content in the source document's
  language.
- How does the pipeline behave when a very short document (< 200 chars) provides too little
  context for the classifier to determine type? It should produce a clear non-KB classification
  rather than guessing.
- What happens if a prompt rewrite causes the model to stop calling a required tool (e.g.,
  `write_kb_entry`)? The pipeline must detect this and surface a warning in the import report.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-000**: **[CRITICAL — SEMANTIC EQUIVALENCE]** Every prompt rewrite MUST preserve the
  original semantic intent and functional behavior. The rewrite changes structure and
  expression only; it MUST NOT add new behaviors, remove existing tool-call steps, alter
  output schemas, or change what the model is instructed to do. Any rewrite that changes
  observable pipeline behavior is a defect, not an improvement.
- **FR-001**: Each prompt component MUST be rewritten as a self-contained, structured document
  with clearly labeled sections (## Role / ## Task / ## Constraints / ## Output Format) plus
  explicit DO/DON'T rule lists and at least one concrete example per KB entry type where
  applicable.
- **FR-002**: Every prompt MUST include an explicit, enumerated list of forbidden behaviors
  (e.g., "DO NOT invent resolution steps not present in the source").
- **FR-003**: The Extractor prompt MUST include a concrete example of a correctly structured
  entry for each KB type, so the model can pattern-match rather than infer structure.
- **FR-004**: The ContentVerifier prompt MUST specify a strict output schema and instruct the
  model to return an empty `verified_fields` list rather than hallucinated confirmations when
  it cannot find source support.
- **FR-005**: The DocumentClassifier prompt MUST enumerate all KB entry types with distinguishing
  characteristics, so the model can differentiate between, e.g., a `process` document and a
  `guideline` document without guessing.
- **FR-006**: All prompts MUST be measurably shorter than their predecessors (fewer input
  tokens) while maintaining or improving output quality.
- **FR-007**: Prompt changes MUST NOT break any existing passing tests in the test suite.
- **FR-008**: The ImportAgentRunner system prompt MUST instruct the model to complete ALL
  drafts passed to it before finishing, with an explicit "Process ALL N drafts" directive.
- **FR-009**: The SkillAdvisor prompt MUST provide explicit thresholds and criteria for SKIP /
  OPTIONAL / RECOMMENDED recommendations rather than relying on vague guidance.
- **FR-010**: Prompt components MUST be improved in priority order: (1) Extractor, (2) Runner
  system prompt, (3) Classifier, (4) Reader, (5) Verifier, (6) SkillAdvisor. Each component
  is independently shippable — later components may be deferred. SkillExecutor is explicitly
  out of scope: it already follows the Claude Code pattern (SKILL.md body = system prompt).

### Key Entities

- **Prompt Component**: A named string constant used as system or user prompt in one pipeline
  stage (Classifier, Reader, Extractor, Verifier, Runner, SkillAdvisor).
- **Prompt Quality Dimension**: One of four measurable axes — instruction-following rate,
  hallucination rate, token count, and tool-call completeness.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: After prompt rewrites, importing 10 reference documents of mixed types produces
  pending entries where 100% have sections matching their declared type (0 section crossover
  failures), up from the current baseline where failures are observed.
- **SC-002**: The content verifier flags at least 90% of deliberately injected fabricated
  fields in a test document set, compared to the current observed rate.
- **SC-003**: Total input token count per standard import run decreases by at least 15%
  compared to the same run before prompt rewrites, with no increase in entry rejection rate.
- **SC-004**: Batch import of 10 same-type documents produces correctly typed entries for at
  least 9 of 10 without any `--type` override.
- **SC-005**: All existing passing tests continue to pass after prompt rewrites — this is the primary gate for semantic equivalence. Any rewrite that causes a previously-passing test to fail is rejected regardless of structural quality.
- **SC-006**: For each rewritten prompt, a reviewer can map every instruction in the new version back to a corresponding instruction in the original version (one-to-one semantic traceability). No net-new behavior is introduced.

## Clarifications

### Session 2026-06-18

- Q: 改写 prompt 时采用哪种具体结构风格？ → A: 分节标题（## Role / ## Task / ## Constraints / ## Output Format）+ 显式 DO/DON'T 列表 + 每种 KB 类型各一个完整示例。**关键约束：改写必须保持语义等价——所有原有行为和功能效果不得改变，仅改变表达方式和结构。**
- Q: 6 个 prompt 组件的改写优先级？ → A: Extractor 优先，其次 Runner system prompt，再其次 Classifier / Reader / Verifier / SkillAdvisor（按此顺序逐步实现）。
- Q: SkillExecutor prompt 是否纳入改写范围？ → A: 排除。SkillExecutor 已采用 Claude Code 模式（SKILL.md body 即 system prompt，executor 透传），无需改写。本次范围为 6 个导入管线组件：Extractor / Runner / Classifier / Reader / Verifier / SkillAdvisor。

## Assumptions

- Prompt rewrites affect only string constants in the codebase; no changes to pipeline
  control flow, tool definitions, or data model are needed.
- The reference to Claude Code's prompt style is for inspiration on structure and clarity
  (labeled sections, explicit DO/DON'T rules, numbered steps, examples) — not for copying
  verbatim content.
- Token count reduction is measured against the Anthropic API's reported input token counts
  for a fixed set of 5 reference import documents.
- Existing test coverage is sufficient to catch regressions; no new integration tests need
  to be added as part of this feature (only unit tests for specific prompt properties).
- The SkillExecutor prompt is lower priority than the other five components and may be
  deferred if time is limited; the other five are considered in-scope for this feature.
