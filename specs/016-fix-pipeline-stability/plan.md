## Summary

Fix seven quality and stability issues found in v14 usage testing of the three-phase import pipeline (feature 015). Five P2/P3 issues (D-3 through D-7) degrade reliability and observability. All fixes are prompt changes, one-line guards, or small code additions within existing files — no new modules needed.

## Implementation Design

### D-1: Extractor Draft Validation + Repair (`extractor.py`, `pipeline.py`)

**`ExtractorAgent._validate_and_repair_draft(draft: str) -> tuple[str, str | None]`**

Validation steps (in order):
1. Strip prose before the first `---` (e.g., "Here is the KB entry:\n---")
2. Check for closing `---` after the YAML block; append if missing
3. Attempt `frontmatter.loads()` on the result; if it raises, return `("", error_msg)`
4. On success: return `(repaired, warning_msg_if_repaired)`

In `pipeline.py` extraction loop, before passing draft to verifier:
```python
repaired, warning = ExtractorAgent._validate_and_repair_draft(draft)
if not repaired:
    report.errors.append(f"kp-{kp.id}: draft format error — {warning}")
    continue  # skip this KP, continue with others
if warning:
    report.warnings.append(f"kp-{kp.id}: draft repaired — {warning}")
```

### D-2: Verbatim Command Instruction (`extractor.py`)

Append to `EXTRACTOR_SYSTEM_PROMPT` (after the existing rules):
```
CRITICAL FOR ## Resolution:
- Copy shell commands VERBATIM from the source text. Do NOT paraphrase or
  summarize commands. Any text inside code fences (```bash ... ```) or
  command lines must be reproduced exactly as written in the source.
- Prose descriptions outside code blocks may be paraphrased.
```

### D-3: KP Scoping Instruction (`reader.py`)

Append to `READER_SYSTEM_PROMPT` (after the existing IMPORTANT RULES):
```
KNOWLEDGE POINT SCOPING:
- One incident = ONE knowledge point. Do NOT create separate KPs for the
  symptoms, root cause, and resolution of the SAME incident.
- A knowledge point represents a complete problem-solution pair.
- Only split into multiple KPs when topics are clearly independent
  (different systems, different time periods, or explicitly separate incidents).
```

### D-4: 0-KP Warning (`pipeline.py`)

In `ThreePhaseImportPipeline.run()`, after `reader.run()` returns:
```python
if len(knowledge_map.knowledge_points) == 0:
    report.warnings.append(
        "No knowledge points identified — document may not contain "
        "incident or runbook content recognizable to the Reader."
    )
```

### D-5: Dedup Step in Extraction Prompt (`pipeline.py`)

In `_run_extraction_loop()`, when building `user_prompt` with kp_drafts, replace the step list:
```
For each draft:
0. Call compare_root_cause with the draft's title and root cause summary to check
   for existing similar entries. If similarity ≥ 0.8, call write_kb_entry with
   update=True to merge rather than creating a new entry.
1. Call verify_content with the draft to check field support ...
2. Call write_kb_entry with the verified content.
3. Call evaluate_skill / create_skill_for_entry if appropriate.
```

### D-6: Skill run.sh Commands (`manager.py`, `tools.py`, `runner.py`)

**`manager.py`**: Add `commands: list[str] | None = None` to `create_skill()`. When commands is non-empty, write them as a script body instead of the placeholder:
```python
if commands:
    cmd_lines = "\n".join(commands)
    run_sh_content = header + f"\n{cmd_lines}\n"
else:
    run_sh_content = generate_run_sh_template(description)
```

**`tools.py`**: Add `resolution_commands` (array of strings) to `create_skill_for_entry` input schema. Pass it to `create_skill()` when calling.

**`runner.py`**: In `_run_skill_and_curation()`, extract commands before calling the tool:
```python
commands = detect_commands(resolution_text)
result = create_skill_for_entry(ctx, {
    "name": advice.suggested_name,
    "entry_id": entry_id,
    "description": advice.reason,
    "resolution_commands": commands,
})
```

### D-7: Last-Write-Wins Trace (`tools.py`)

In `write_kb_entry()` (or wherever `DecisionTrace` is mutated), enforce mutual exclusion:
```python
# When marking a field as verified:
trace.field_sources[field] = fragment
trace.unsupported_fields = [f for f in trace.unsupported_fields if f != field]

# When marking a field as cleared:
trace.unsupported_fields.append(field)
trace.field_sources.pop(field, None)
```

## Execution Order

```
Phase 1 (Setup):     None — no new files to create
Phase 2 (P1 fixes):  D-2 (prompt, no deps) → D-1 (code, needs frontmatter understanding)
Phase 3 (P2 fixes):  D-3 (prompt), D-4 (code), D-5 (prompt), D-6 (code, 3 files)
Phase 4 (P3 fix):    D-7 (code)
Phase 5 (Tests):     Extend existing test files; run full pytest
```

D-2 before D-1: fixing the prompt reduces malformed drafts, making repair logic less exercised in practice but still necessary for robustness.

D-3, D-4, D-5 are independent and can be done in any order.

D-6 spans 3 files (`manager.py` → `tools.py` → `runner.py`) and must be done sequentially in that order (bottom-up dependency).
